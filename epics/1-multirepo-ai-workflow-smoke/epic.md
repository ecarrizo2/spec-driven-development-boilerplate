# Epic: Multi-Repo AI Workflow Smoke Test

## Problem Statement

The automation chain needs a clean, deterministic test run from epic approval through AI-generated plan synthesis and child repo dispatch. We need a small multi-repo epic that verifies orchestration works end-to-end without feature-level complexity.

## Product Vision

After approval, the system should automatically create plan PRs, synthesize plan content with AI, enforce dependency order, and dispatch execution to the target repo so the child execution PR is ready for engineering handoff.

## Requirements

- **R1.** The epic must include one API task and one frontend task, with the frontend task depending on the API task.
- **R2.** Approving and merging the epic PR must create draft plan PRs for both tasks.
- **R3.** Each plan PR must run the AI synthesis workflow and publish the `Plan Generated by Agent` check.
- **R4.** Merging the first approved plan PR must dispatch the task to `sdd-test-api`; merging the second approved plan PR must dispatch to `sdd-test-frontend` only after dependency conditions are satisfied.
- **R5.** Delivery tracking metadata must be updated so the run is auditable.

## Success Criteria

- [ ] WHEN epic PR # is approved and merged, the system SHALL create two plan PRs linked to task IDs 1 and 2.
- [ ] WHEN each plan PR is opened, synchronized, or marked ready for review, the system SHALL complete a successful `Plan Generated by Agent` check run.
- [ ] IF task 1 is not completed, the system SHALL mark task 2 plan dependency validation as blocked.
- [ ] WHEN task 1 plan PR is merged, THEN a child execution PR is created in `ecarrizo2/sdd-test-api`.
- [ ] WHEN task 2 plan PR is merged after task 1 completion, THEN a child execution PR is created in `ecarrizo2/sdd-test-frontend`.

## Scope Boundaries

### In Scope

- Validate orchestration behavior for epic approval, plan generation, AI pickup, dependency gating, and dispatch.
- Create minimal request definitions sufficient for plan generation.

### Out of Scope

- Implementing production features in API or frontend repos.
- Performance/load testing of workflow throughput.
- Jira workflow customization beyond default ticket creation behavior.

## Repos Involved

| Repo | Role in this epic | Type of changes |
|------|-------------------|-----------------|
| `sdd-test-api` | First dispatch target | Execution branch + draft PR creation |
| `sdd-test-frontend` | Second dispatch target (depends on API task) | Execution branch + draft PR creation |

## Technical Constraints

- Task graph must remain in YAML frontmatter format consumable by `readTaskGraphTasks`.
- Request file naming must match plan creation lookup behavior from `epic-create-plans.yml`.
- Dependency metadata in task graph must be consistent with delivery node dependencies.
- Child repos must continue using reusable `receive-task.yml` caller workflows.

## Cross-Repo Dependencies

- Task 2 (`sdd-test-frontend`) depends on task 1 (`sdd-test-api`) to validate dependency gating and ordered dispatch.
- Delivery merge order should place frontend PR node after API PR node.

## Decisions Made During Discovery

| # | Question | Decision | Rationale |
|---|---|---|---|
| 1 | Should this be a one-task or multi-repo test? | Multi-repo test selected | User requested both repos be exercised. |
| 2 | How to keep test fast while proving dependencies? | Use 2 tasks with one dependency | Smallest path that validates dependency guardrails. |
| 3 | Which repos to include? | `sdd-test-api` and `sdd-test-frontend` | Matches configured managed test repos. |

## Definition of Done

- [ ] **Deployment:** Epic and plan automation flows complete for both tasks and dispatch execution PRs in child repos.
- [ ] **Monitoring:** Workflow runs and check runs visible in GitHub Actions for epic/plan/dispatch stages.
- [ ] **Sign-offs:** Human approves epic PR and both plan PRs.
- [ ] **Documentation:** Delivery manifest records child PR URLs for traceability.
- [ ] **Success metrics:** 100% of expected automation stages succeed in this test run.
- [ ] **Manual steps:** No manual delivery steps required for this smoke test.
